Closes #862

## Summary

Implements HD wallet derivation and configurable address rotation for all five pipeline agents (Scout, Ledger, Signal, Scribe, Executor), preventing long-term address clustering and observer correlation.

## Problem

Previously each agent used a single permanent funded testnet wallet (e.g. `SCOUT_SECRET_KEY`). Any observer watching that one address could correlate all Scout transactions across all users indefinitely.

## Solution

### HD Wallet Derivation (`agent-hd-wallet.ts`)
- Single BIP-39 mnemonic (`AGENT_MASTER_SEED`) stored as a secret ‚Äî never logged or persisted
- SLIP-0010 Ed25519 derivation path `m/<role_index>'/<address_index>'` so each agent's namespace is fully isolated
- Deterministic: same seed + same indices always produce the same keypair (safe to re-derive without storing private keys)

### Address Rotation Service (`agent-wallet-rotation.ts`)
- Configurable via `AGENT_ROTATION_SCHEDULE`: `per-task` | `daily` (default) | `weekly` | `manual`
- On rotation: derives next keypair, drains XLM from old address ‚Üí new address via Stellar payment tx, persists new state to DB
- Drain is non-fatal ‚Äî rotation completes even if the on-chain drain fails (logged, null `drain_tx_hash`)
- In-memory keypair cache for daily/weekly modes to avoid re-deriving on every request

### Database Migration (`024_create_agent_wallet_tables.sql`)
- `agent_wallet_rotations` ‚Äî one row per agent, tracks `current_index`, `public_key`, `last_rotated_at`, `rotation_count`
- `agent_wallet_history` ‚Äî immutable append-only audit log of every address ever used, with drain tx hash
- RLS: service_role only (no user-facing access)
- `increment_rotation_count` stored function for atomic counter updates

### Admin REST API (`GET/POST /api/admin/agent-wallets`)
- `GET  /api/admin/agent-wallets` ‚Äî list rotation state for all agents
- `GET  /api/admin/agent-wallets/:agent` ‚Äî state + full address history
- `POST /api/admin/agent-wallets/rotate` ‚Äî force-rotate all agents
- `POST /api/admin/agent-wallets/:agent/rotate` ‚Äî force-rotate one agent
- Protected by existing `adminAuth` middleware + rate limiter

### Scheduler Integration (`scheduler.ts`)
- Daily cron at `00:05 UTC` ‚Äî checks and rotates if schedule is due
- Weekly cron on Mondays at `00:10 UTC` ‚Äî belt-and-suspenders for weekly schedule

### Config
- `AGENT_MASTER_SEED` (optional string) ‚Äî BIP-39 mnemonic
- `AGENT_ROTATION_SCHEDULE` (enum, default: `daily`)
- Both documented in `.env.example` with generation instructions
